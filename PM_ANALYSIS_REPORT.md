# Product Manager Analysis Report

**Analyst**: Senior PM (Silicon Valley)
**Date**: 2026-01-13
**Product**: RAI - AI Resume Intelligence Platform
**Target User**: 헤드헌터 (Executive Recruiters)

## Executive Summary

QA 리포트 분석 결과, **기술적 우선순위와 비즈니스 우선순위가 일치하지 않는 케이스**가 다수 발견됨.
헤드헌터의 실제 워크플로우 관점에서 재평가하면, 일부 "CRITICAL" 버그는 실제로 발생 확률이 낮고,
반면 "LOW" 등급의 일부 이슈는 **일상 업무에 직접적인 마찰**을 일으킬 수 있음.

### TL;DR - 비즈니스 관점 우선순위

| Rank | Issue | QA 등급 | PM 등급 | 이유 |
|------|-------|---------|---------|------|
| 1 | Mixed Language Query 미지원 | LOW | **CRITICAL** | 한국 헤드헌터가 매일 사용하는 패턴 |
| 2 | Cache Stampede | CRITICAL | HIGH | 실제 동시 접속 1000명 미만 예상 |
| 3 | Facet 빈 문자열 스킬 | LOW | **HIGH** | 데이터 품질 이슈로 사용자 신뢰 저하 |
| 4 | Network Flapping 다중 콜백 | MEDIUM | **HIGH** | 모바일/카페 사용자 경험 직결 |
| 5 | localStorage 알림 없음 | MEDIUM | MEDIUM | 파워유저만 영향 |
| 6 | Integer Overflow | CRITICAL | **LOW** | 21억 회 사용 비현실적 |

## Part 1: User Journey 기반 버그 재평가

### 헤드헌터의 하루 (Typical User Journey)

```
09:00 - 출근, 어제 저장한 검색 불러오기
09:30 - 새로운 JD 받음, "React 5년차 핀테크" 검색
10:00 - 후보자 20명 리스트업, 필터로 좁히기
11:00 - 개별 후보자 상세 리뷰, 리스크 체크
14:00 - 클라이언트 미팅 (모바일로 후보자 확인)
16:00 - 비슷한 이름 중복 체크 (홍길동 vs 홍길돈)
17:00 - 검색 저장하고 퇴근
```

### Journey별 버그 영향도 분석

#### 1. 검색 단계 (09:30) - **가장 빈번한 액션**

| Bug | 발생 확률 | 사용자 영향 | 비즈니스 손실 |
|-----|----------|------------|--------------|
| **Mixed Language Query** | 95% | 검색 결과 누락 | **매우 높음** |
| Skills Array Null | 5% | 검색 실패 | 중간 |
| Cache Stampede | <1% | 느린 응답 | 낮음 |

**PM 의견**:
> "React 개발자 5년" 같은 **한영 혼합 쿼리**는 한국 헤드헌터의 기본 검색 패턴입니다.
> 이걸 지원 안 하면 사용자가 불편함을 느끼고 **수동으로 쿼리를 나눠서 검색**해야 합니다.
> 이건 LOW가 아니라 **DAY 1 FIX** 대상입니다.

#### 2. 필터링 단계 (10:00)

| Bug | 발생 확률 | 사용자 영향 |
|-----|----------|------------|
| Facet 빈 문자열 스킬 | 30% | 이상한 빈 태그 표시 |
| Synonym Expansion Overflow | 10% | 느린 필터링 |
| 0 count 필터 표시 | - | 이미 해결됨 (PRD P1) |

**PM 의견**:
> Facet에 빈 문자열이나 공백만 있는 스킬이 보이면 **"이 시스템 데이터 괜찮나?"** 의구심.
> 신뢰도 문제는 헤드헌터에게 치명적 - 클라이언트에게 추천할 데이터니까요.

#### 3. 모바일 확인 (14:00) - **성장하는 Use Case**

| Bug | 발생 확률 | 사용자 영향 |
|-----|----------|------------|
| Network Flapping 다중 콜백 | 40% | 데이터 중복 로드, 배터리 소모 |
| onReconnect 무한 호출 가능성 | 10% | 앱 성능 저하 |

**PM 의견**:
> 카페나 이동 중 검색하는 헤드헌터가 늘고 있습니다.
> 불안정한 WiFi에서 **화면이 깜빡거리거나 로딩이 반복**되면 바로 이탈합니다.
> 이건 MEDIUM이 아니라 **모바일 UX의 핵심**입니다.

#### 4. 중복 체크 (16:00)

| Bug | 발생 확률 | 사용자 영향 |
|-----|----------|------------|
| pg_trgm 공백 이름 | <1% | 에러 발생 |
| 유사 이름 threshold 설정 | - | 스펙 부재 |

**PM 의견**:
> 유사 이름 검색은 잘 작동하지만, **사용자가 threshold를 조절할 UI가 없습니다**.
> 이건 버그가 아니라 **스펙 누락**입니다.

## Part 2: 스펙아웃 필요 항목

### SPEC-001: Mixed Language Query Support

**현재 상태**: "React 개발자" → 하나의 토큰으로 처리 → 결과 없음
**예상 동작**: "React" + "개발자" 분리 → 각각 동의어 확장 → 결과 반환

```
User Story:
- AS A 헤드헌터
- I WANT TO 한영 혼합으로 자연스럽게 검색
- SO THAT 별도의 쿼리 조합 없이 빠르게 후보자를 찾을 수 있다

Acceptance Criteria:
1. 공백으로 구분된 토큰 각각에 동의어 확장 적용
2. 한글 토큰은 한글 동의어, 영문 토큰은 영문 동의어
3. 쿼리 자동 제안에 혼합 패턴 포함
```

### SPEC-002: Similar Name Threshold UI

**현재 상태**: API에서만 threshold 파라미터 지원, UI 없음
**예상 동작**: 슬라이더 또는 프리셋으로 유사도 조절

```
User Story:
- AS A 헤드헌터
- I WANT TO 이름 유사도 민감도를 조절
- SO THAT 동명이인 체크 시 너무 많거나 적은 결과를 피할 수 있다

Proposed UI:
- 프리셋: "정확한 매칭" (0.9) / "유사 이름 포함" (0.5) / "폭넓게" (0.3)
- 고급 옵션: 슬라이더 0.0 ~ 1.0
```

### SPEC-003: Search History Cross-Tab Sync

**현재 상태**: 탭 간 히스토리 동기화 안 됨
**예상 동작**: 다른 탭에서 검색해도 실시간 반영

```
User Story:
- AS A 헤드헌터
- I WANT TO 여러 탭에서 작업해도 검색 기록이 동기화
- SO THAT 어느 탭에서든 최근 검색을 이어갈 수 있다

Implementation:
- BroadcastChannel API 또는 storage event 리스닝
```

### SPEC-004: Bulk Action Progress Indicator

**현재 상태**: 일괄 작업 시 전체 진행률만 표시
**예상 동작**: 개별 항목 처리 상태 표시

```
User Story:
- AS A 헤드헌터
- I WANT TO 일괄 삭제 시 어떤 항목이 처리되었는지 확인
- SO THAT 실패한 항목을 개별적으로 재시도할 수 있다

Proposed UI:
- 각 항목에 체크마크/X 표시
- 실패 시 "재시도" 버튼
- 전체 진행률 + 성공/실패 카운트
```

### SPEC-005: Offline Mode with Queue

**현재 상태**: 오프라인 시 에러 표시만
**예상 동작**: 오프라인 액션 큐잉 후 온라인 복구 시 자동 처리

```
User Story:
- AS A 모바일 헤드헌터
- I WANT TO 오프라인에서도 메모/태그 작업
- SO THAT 지하철에서도 후보자 정리를 계속할 수 있다

Scope:
- Phase 1: 읽기 전용 캐시 (이미 본 후보자 정보)
- Phase 2: 쓰기 작업 큐잉 (메모, 태그)
```

## Part 3: 추가 개선 권장 사항

### 헤드헌터 Pain Point 기반 개선안

#### 1. 검색 결과 "왜 이 후보자가 나왔는지" 설명

**현재**: matchScore만 표시
**개선**: 매칭 이유 하이라이트

```
예시:
- "React 5년" 검색 시
- 후보자 카드에: "✓ React 6년 경력" / "✓ 핀테크 업계"
```

**비즈니스 가치**: 클라이언트에게 왜 이 후보자를 추천하는지 설명하기 쉬움

#### 2. "내가 이미 본 후보자" 필터

**현재**: 없음
**개선**: 조회 기록 기반 필터

```
- "이미 본 후보자 숨기기" 토글
- "지난 7일간 본 후보자" 필터
```

**비즈니스 가치**: 중복 작업 방지, 시간 절약

#### 3. 검색 비교 기능

**현재**: 검색 결과는 각각 독립
**개선**: 두 검색 결과 비교 뷰

```
- "React 개발자" 검색 결과 vs "Vue 개발자" 검색 결과
- 겹치는 후보자 하이라이트
- "두 스킬 모두 보유" 필터
```

**비즈니스 가치**: 다중 포지션 운영 시 효율성

#### 4. 후보자 "온도" 지표

**현재**: 정적 데이터만
**개선**: 시장 활동 지표

```
- 최근 이직 시점
- 예상 이직 의향 (LinkedIn 활동 기반?)
- 유사 프로필 채용 트렌드
```

**비즈니스 가치**: "지금 접촉해도 될 후보자인가" 판단 지원

## 최종 권장사항

1. **즉시 실행**: Mixed Language Query 지원 - 이건 버그가 아니라 **핵심 기능 누락**
2. **데이터 품질**: Facet 빈 문자열 이슈는 UI뿐 아니라 **데이터 정제 파이프라인** 점검 필요
3. **모바일 집중**: Network 관련 이슈는 **모바일 사용자 증가**에 대비한 선제 대응
4. **스펙 문서화**: Similar Name Threshold 같은 "숨은 기능"은 **사용자 가이드** 또는 **UI 노출** 필요
5. **메트릭 우선**: 버그 수정 전에 **현재 사용 패턴 데이터** 수집 → 우선순위 검증
